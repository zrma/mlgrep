# v0.2 Streaming Memory Evidence

Status: implementation evidence; release pending

## Method

The workload uses deterministic 128-byte LF-terminated lines and exact UTF-8
literal count search. Each current measurement is the median of three native
runs on macOS arm64. `/usr/bin/time` supplies per-process peak RSS. These are
observations, not portable performance guarantees.

The canonical current command is:

```sh
python3 scripts/measure-streaming-memory.py \
  --binary target/mlgrep \
  --output target/streaming-memory.json \
  --enforce-bounded
```

## Results

| Input | v0.1 whole-file median time | v0.1 peak RSS | v0.2 streaming median time | v0.2 median peak RSS |
| ---: | ---: | ---: | ---: | ---: |
| 1 MiB | about 0.03 s | about 3.8 MiB | 0.019 s | 1.33 MiB |
| 10 MiB | about 0.32 s | 24.6-26.6 MiB | 0.129 s | 1.31 MiB |
| 100 MiB | 3.27-3.30 s | about 233.5 MiB | 1.227 s | 1.31 MiB |

At 100 MiB, observed peak RSS fell by about 99.4%. More importantly, current
peak RSS did not grow with total input size in this workload.

## Gate

The cross-platform check fails if any current run exceeds 32 MiB peak RSS or if
peak RSS grows by more than 16 MiB from the smallest to largest input. These
ceilings deliberately leave host variance while detecting a return to
whole-file retention. Correct output is checked on every measured run.
